가입·인증 UI 부재 의도

  • 사용자가 회원가입을 시도할 경로 자체가 없다. 회원가입·비밀번호 재설정 라우트 미배포.
  • /admin 라우트는 미들웨어로 보호. auth.uid()가 admin_users에 없으면 404 응답 (401 아님 — admin 존재 자체를 숨김).
  • 대표 본인이 비밀번호 분실 시 Supabase Studio에서 직접 재설정.
  • Phase 1에서 일반 회원 도입 시점에 가입·로그인 UI 신설 + 약관·개인정보처리방침 동시 배포.
왜 이 모델인가

대표 통찰 — "단가 계산은 로그인 없이 화면에서 하는 것이고, CMS는 나만 보면 된다." 이 한 마디로 약관·개인정보 수집·승인 워크플로·역할 매트릭스·이메일 인프라가 모두 Phase 1 이후로 미뤄졌다. Phase 0 구현 분량 체감 30~40% 감소. Karpathy 2원칙(Simplicity First) 완벽 부합.

create function is_super_admin() returns bool
language sql security definer stable as $$
  select exists (
    select 1 from admin_users
    where id = auth.uid() and role = 'super-admin'
  );
$$;

create function log_admin_change() returns trigger
language plpgsql security definer as $$
begin
  insert into audit_log(actor_id, action, table_name, row_id, before_json, after_json)
  values(
    auth.uid(), TG_OP, TG_TABLE_NAME,
    coalesce(NEW.id, OLD.id),
    case when TG_OP = 'INSERT' then null else to_jsonb(OLD) end,
    case when TG_OP = 'DELETE' then null else to_jsonb(NEW) end
  );
  return coalesce(NEW, OLD);
end; $$;

S07

단가 계산 로직

CORE LOGIC

본 섹션이 제품의 본질이다. 대표가 강조한 "각 인원별 단가가 중요"는 이 계산 모델로 구현된다. 모든 계산은 클라이언트 단독으로 수행하며 서버 호출이 없다.

입력 모델 (★ 2026-05-17 — productType + longHaul 추가)

type ProductType =
  | 'half-day'      // 반나절 (≤4h, Day 없음, 시간 단위)
  | 'one-day'       // 1일 투어 (Day 1, 숙박 X)
  | 'one-night'     // 1박 2일
  | 'two-night'     // 2박 3일
  | 'three-night'   // 3박 4일
  | 'four-night'    // 4박 5일
  | 'five-night'    // 5박 6일 (장기 경계)
  | 'long-haul';    // 6박+ 장기 (별도 빌더)

type CalculationInput = {
  productType: ProductType;        // ★ 빌더 진입 첫 선택
  durationHours?: number;          // half-day: 2~4 시간
  nights?: number;                 // one-night~five-night: 1~5, long-haul: 6+
  party: { adult: number; child: number; infant: number; senior: number };
  ageTierMultipliers: { adult: number; child: number; infant: number; senior: number };
  days: DayPlan[];
  partyShared: PartyCost[];            // 차량·가이드 등 파티 공통 비용
  season: { multiplier: number; appliesTo: string[] };
  margin: { net: number; retail: number };
  vatRate: number;
  exchangeRate: { krwToUsd: number };
};

type DayPlan      = { dayNumber: number; items: PerPersonItem[] };
type PerPersonItem = { categorySlug: string; placeId?: string; unitPriceKrw: number; applyAgeTier: boolean };
type PartyCost    = { categorySlug: string; totalKrw: number; notes?: string };

계산식 (단계별)

STEP 1 — 가중 인원 (N_eff) N_eff = adult × m_adult + child × m_child + infant × m_infant + senior × m_senior // 예: 성인 18 + 소아 2 = 18×1.0 + 2×0.5 = 19.0 STEP 2 — 시즌 보정 seasonCost(category, cost) = cost × (season.appliesTo.includes(category) ? season.multiplier : 1.0) STEP 3 — 파티 공통 비용의 1인당 분배 partySharedPerPerson = Σ seasonCost(p.categorySlug, p.totalKrw) / N_total // N_total = 실제 머릿수 (가중치 적용 전) STEP 4 — 연령별 인당 항목 합계 perPersonItemsForTier(t) = Σ seasonCost(item.cat, item.unitPriceKrw) × (item.applyAgeTier ? m_t : 1.0) STEP 5 — Tier 원가 tierCost(t) = perPersonItemsForTier(t) + partySharedPerPerson STEP 6 — 마진 netPrice(t) = tierCost(t) × (1 + margin.net) retailPrice(t) = netPrice(t) × (1 + margin.retail) STEP 7 — VAT 적용 최종 1인당 finalKrw(t) = retailPrice(t) × (1 + vatRate) finalUsd(t) = finalKrw(t) × exchangeRate.krwToUsd STEP 8 — 그룹 총액 groupTotalKrw = Σ_t (party[t] × finalKrw(t)) STEP 9 — 채널별 수수료 차감 (★ 2026-05-17 추가) channelFee(c) = finalKrw(adult) × c.commission_rate cardFee(c) = finalKrw(adult) × c.card_fee_rate + c.fixed_fee_krw netRevenuePerAdult(c) = finalKrw(adult) − channelFee(c) − cardFee(c) // 자체 오프라인: 수수료 0 · 자체 온라인: 카드 2.8% · B2B: 10~20% STEP 10 — 채널별 1인당 순수익·수익률 profitPerAdult(c) = netRevenuePerAdult(c) − tierCost(adult) profitRate(c) = profitPerAdult(c) / finalKrw(adult) × 100 // 수익률은 판매가 대비. tierCost 대비가 아님. 영업 관점 표준. STEP 11 — 인원 N별 수익 매트릭스 (★ What-if 핵심) forEach N in [5,10,15,20,25,30,35,40,45]: N_total' = N (실제 머릿수) partyShared' = Σ partyShared (시즌 보정 적용) / N tierCost'(adult) = perPersonItemsForTier(adult) + partyShared' netRevenue'(c) = finalKrw(adult) × (1 − c.commission_rate − c.card_fee_rate) − fixed_fee profitRate'(N, c) = (netRevenue'(c) − tierCost'(adult)) / finalKrw(adult) × 100 // 결과: 5 채널 × 9 인원 구간 = 45 셀 매트릭스 STEP 12 — 손익분기점·의사결정 신호 breakEvenN(c) = min N s.t. profitRate'(N, c) >= 0 bestChannelAt(N) = argmax_c profitRate'(N, c) marginalProfit(c) = profitRate'(N+5, c) − profitRate'(N, c) // "5명 더 받으면 수익률 +N%p" 즉시 표시

  • B2B 소매 BEP: 35명.

왜 이게 핵심 가치인가

단가 계산만 하는 도구는 시장에 흔하다(엑셀). 본 제품의 차별 가치는 "이 인원·이 채널로 팔면 흑자인가, 몇 명을 더 채워야 흑자인가, 자체 온라인으로 직판하는 게 B2B 소매보다 얼마나 이익인가"를 즉답한다는 것. 통합기획서 [docs/통합기획서.html:1672](통합기획서.html#L1672) "What-if 차량 비교"·[docs/통합기획서.html:1679](통합기획서.html#L1679) "커미션 체인 4단계"의 합성으로, 본 제품의 본질을 이 한 표가 정의한다.

계산 정밀도 규칙
  • 중간 계산은 JavaScript Number (IEEE 754 double) 그대로 사용 — 통화 합산 ~20조 KRW까지 안전.
  • 최종 출력 시 Math.round로 1원 단위 반올림. USD는 0.01 단위.
  • 표시 포맷: Intl.NumberFormat('ko-KR') · Intl.NumberFormat('en-US', { currency: 'USD' }).
  • 음수 입력 차단 — react-hook-form + zod 스키마에서 z.number().nonnegative().

  2. "각 인원별 단가"가 핵심 가치라면, 본 문서의 와이어프레임은 정량적 검증 부족. 결과 화면(P02)에 인원별 4구간 카드를 강조했으나, 실제 사용자가 "성인 178k" 한 숫자만 보는지·4구간을 모두 활용하는지는 유저 테스트 전엔 미지수.
  3. RLS 정책의 단순성이 미래 부담을 숨긴다. Phase 0은 "마스터 read-public + super-admin write"로 끝나지만, Phase 1 회원 도입 시 사용자별 패키지 저장이 추가되면 정책이 30줄 이상으로 확장. 본 문서는 그 마이그레이션 경로를 명시하지 않음.
